GitHub Repository Forking

Forking Overview

  • Forking a GitHub repo creates a personal copy of an existing project in your personal Github account.

  • A forked repo provides an isolated workspace for contributing to a project without affecting or being dependent on others.

  • This isolation lets you test for conflicts when merging a branch into main before creating a pull request in your production repo.

  • Before testing your branch, you will want to update the main branch of your fork by pulling in the main branch of your production repo (covered in the next section).

  • For this portion of my demo I will be using GitHub's web interface (not command-line).

  • Note, before you can fork a repo you need a GitHub account.

Setting Up Your Local Workspace

Setup Overview

  • Before you can contribute to your project you need to setup your local workspace.

  • Setup a working directory for managing your project.

  • Navigate to your new directory and locally clone your forked repo there.

  • Setup your local repo to communnicate with Github. Your forked project is called the 'origin' repo and your production code is called the 'upstream' repo.

  • Make a new working branch for developing your contributions.

  • For this portion of my demo I will only be using the windows command-line interface.

  • Note, for demo purposes I start working in the default directory but in practice its a good idea to establish an organized directory ahead of time.

Setting Up & Navigating Your Repository Directory

Step: 1

To also connect your local repo with your production repo (called the upstream), use the 'add new remote' command:

Local Code Development Using Git Version Control

Coding Overview

  • I will explore my local workflow for managing code edits and keeping my work in sync with GitHub by creating an html feature to contribute to my teams production code.

  • Throughout development you will want to regularly keep your local repo in sync with your production code.

  • Throughout development you will also want to regularly test your code (which I will explore in the testing section).

  • You will want to adopt an iterative development approach when working with Git. Meaning, you will work on the task of implementing a feature as a series of smaller sub-tasks.

  • Your working branch represents the feature you want to add to your production code (so it's a good idea to give your branch a descriptive name).

  • Use Git to manage your branch by adding and commiting your code edits whenever you complete a sub-task. Commiting code keeps a working record of your contributions.

  • Sync your branch with your origin repo to back up your work online when your ready.

  • Before I explore how to incorporate Git into your workflow, I will show you what my working desktop looks like.

Keeping Your Local Repo Updated

Step: 1

First, switch back to your main branch.

To make sure your main branch is up to date with your production code you will need to fetch and merge your upstream repo into your origin repo using the 'pull' command:

git pull upstream main

(image right)

Step: 2

Then, switch back to your feature branch.

To make sure your feature branch is up to date with your production code you can merge your main branch into your feature branch using the 'merge' command:

git merge main

(image right)

Note:

Another way to check what branch your on is to use the 'staus' command:

git status

This command is also useful to see if you have uncommited changes that can be added (demonstrated in the next section).

Adding, Commiting, & Pushing Code With Git.

Step: 1

The first step is to create a code edit like the demo above (which represents a component of the feature your working on).

To verify that you have uncommited code you can use the 'status' command:

git status

(image right)

Step: 2

Before you can commit your code you need to add it to a 'staging area' using the 'add' command:

git add -A

(image right)

Step: 3

Next, you will take a working snapshot of the changes you made (which represents a safe version of your project). Everytime you take a snapshot you will document it with a brief descriptive message.

To take a snapshot of your project use the 'commit' command:

git commit -m "[your message here]"

(image right)

Step: 4

After you have commited a series of changes to your branch, back up your work online in your origin repo by using the 'push' command:

git push -u origin [your branch name]

(image right)

Local Testing In Browser & With Super Linter

Testing Overview

  • You will want to simultaneously test your code throughout the development process.

  • Every time you make and save code edits in your editor, refresh your browser to preview your changes, and fix your code if the results are not as expected.

  • If your unsure why your code is not working as intended, you should use the developer tools built into your browser for debugging.

  • Once you finish coding your feature you can locally run your code through super linter (with the same settings as your production repo) to verify that your code passes all required tests before submitting a pull request (so you don't have any surprises).

  • Depending on the type of project your working on you might need to incorporate additional testing into your workflow.

Merging Code Into Your Production Repo With GitHub CLI

Merging Overview

  • If you want to avoid context switching in your workflow, you can create a 'pull request' to merge your working branch with your main production branch, all from the command-line terminal.

  • To do this you will first need to setup Github CLI.

  • GitHub CLI is a open source first party tool for managing your online repo from the terminal.

Creating A Pull Request From Your Terminal

Step: 1

To create a pull request from the terminal use the 'pull request' command:

gh pr create
--title "[Pull Request title]"
--body "[Pull Request Description]"

Next, select where to send your pull request (origin or upstream)
