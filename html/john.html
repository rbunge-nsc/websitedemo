What I Learned this Quarter

My main objective for taking this course was to become confident with git. As someone who has been doing software development for a few years now but never professionally, git had always seemed pretty daunting. It made sense for my own projects, but for those I would only scratch the surface of what git was used for.

In this document I'll outline some of the more interesting things I noticed over my journey learning git this quarter.

Most Helpful Resources

There were two invaluable resources that I used during this course - the official git docs and the Head First Git book from O'Reilly.

The official git docs are the primary source of information about how the tool works. It covers all aspects of the software, and does so in a very condensed and information-dense manner. This works for and against it as a resource. While it is the most thorough, it can also be difficult to understand due to its density. I found that this resource worked best when you had a specific idea of what it was you needed clarification with. If I was banging my head against a wall due to merge conflicts , the official docs were oftentimes the fastest way to an answer.

Using the docs can be more challenging when you want a simplified, broader view of how git works. For these scenarios, I used the more friendly Head First Git.

Head First Git is another book in the O'Reilly "Head First" series, which are slightly less intimidating than their standard offerings. I've found the book series helpful for learning denser topics such as Data Structures and Algorithms in other courses of my study. They are filled with real world examples and abstractions to try and make the more complicated parts of the topic seem more approachable.

Head First Git by Raju Gandhi is no exception to this - I found that when I was really confused on a topic, this book was often the simplest resource to help cut through and build understanding.

This was also very useful for giving me a broader picture of how git works internally.

What I Learned About How Git is Structured

The first thing that surprised me - which maybe shouldn't have, seeing as git was made by Linux creator Linus Torvalds - was that a git repository is simply a series of directories and files.

Every project directory that has been initialized as a git repo has a hidden .git directory created within it to store everything that git needs to function. This can be explored from the root directory of any project that is using git.

What really blew my mind is that this is where git keeps track of the actual changes between branches in the form of diff files. Git works on the files in the directory, and when you switch from branch to branch, it applies all of diff files in order to get the working directory into the state of what it is on any given branch.

This also introduced the important concept of staging area and what really happens with commits. When you make changes to the directory that aren't yet added and committed, there isn't any record of them in the internal git directory. If you were to switch branches without adding and committing these changes, git will completely discard them when it modifies the files to resemble the state of the other branch. This is why it is important to codify finished changes with a git commit and save unfinished changes with git stash.

What I Learned About Branches

The other thing that really surprised me about git was that branches are just commits with a name attached to them. You can see this verified if you dig around in the .git directory.

As you can see when inspecting the jao-font-style-error file in .git/refs/head, the branch itself is just the unique id of the most recent commit. This can be verified by using git log -1 jao-font-sytle-error to view the most recent commit on that same branch.

Up until this point, I had been assuming that branches were more involved and therefore more precious. Branching and managing a workflow between branches was one of the more confusing aspects of git for me at the start of this course. The realization that branches themselves were just references to the most recent commit helped to demystify the concept of branching for me, and also helped me understand the more confusing aspects of branching like solving merge conflicts via rebasing.

What I Learned About Github Actions

The most interesting thing I was able to do as a result of this course was to use Github Actions to automate the deployment of project code. This is something that I had heard about from other devs and had seen implemented in repositories, but had no idea how to do myself.

After learning about Github Actions in class, I explored this concept further and looked at ways that I could use this for my own web development work, and created a repository for a static site that is deployed automatically when code changes are pushed or merged to main.

I compiled my notes for this procedure on github pages. The site itself is managed by a git repository that uses a node module called 11ty to build assets into static resources to be served from a web browser.

It surprised me how simple all of this really was - again, that's been the refrain for using git this quarter. I have already begun using this exact approach for my personal website and will probably using something similar for future projects.

In Summary

The biggest takeaway for me this quarter is that while it can be complex in its usage, git is remarkably simple at the core. It is built on the same principles of representing entities as files that Unix-like operating systems like Linux, macOS, or BSD use internally. Knowing this, I'm able to see more clearly how operations on the git repository itself map to changes in the filesystem and where visual git tools pull their information from.

I have already noticed what I've learned in this course improve my workflow in both my school and personal projects. During my Mobile Application Practicum course in the Application Development program, our project had us working on the open source AnkiDroid App for Android. Jumping into a large, existing project with a very well established repository structure would have been an impossible task, but the knowledge gained from this course helped me successfully contribute to this project.

I'm sure that I will be using git for many years to come over the course of my new career. With how much I've learned this quarter I'm able to appreciate how much more I still have to learn about the tool, and I'm excited to figure out more of its inner workings as I move forward professionally.
